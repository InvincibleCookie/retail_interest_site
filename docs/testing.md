# Автоматическое тестирование сервиса Retail Interest Zones Web App

Документ можно использовать как раздел дипломной работы. Акцент сделан не на пользовательских интервью, а на автоматической проверке сервиса: загрузка одиночного видео, загрузка ZIP-архива с видео, получение кадра, запуск обработки, формирование отчета, скачивание ZIP-отчета и негативные сценарии.

## 1. Цель тестирования

Проверить, что серверная часть приложения корректно принимает входные файлы, создает задания, извлекает видео из ZIP-архивов, возвращает первый кадр, запускает обработку, формирует отчетные файлы и устойчиво обрабатывает некорректные входные данные.

## 2. Объект тестирования

- Backend API на FastAPI: создание задания, загрузка файлов, получение первого кадра, запуск обработки, получение статуса, получение JSON-отчета и скачивание ZIP-отчета.
- Сервис хранения: безопасные имена файлов, фильтрация расширений, распаковка ZIP без выхода за каталог задания.
- Сервис отчетов: интерпретация зон интереса, генерация CSV, Excel, HTML и ZIP.
- Интеграционная связка «загрузка → ROI → обработка → отчет».
- Негативные сценарии: неподдерживаемый файл и ZIP-архив без видео.

## 3. Папка для тестовых данных

Для автоматической проверки создана отдельная папка `service_test_assets/`.

```text
service_test_assets/
├── videos/      # сюда нужно положить одно или несколько настоящих видео
├── archives/   # сюда нужно положить ZIP-архив с одним или несколькими видео
└── invalid/     # необязательно: можно положить неподдерживаемые файлы для ручных негативных проверок
```

Минимально нужно добавить:

1. В `service_test_assets/videos/` — короткое видео формата `.mp4`, `.avi`, `.mov`, `.mkv` или `.m4v`.
2. В `service_test_assets/archives/` — ZIP-архив, внутри которого есть хотя бы одно видео поддерживаемого формата.

Рекомендуемые имена файлов для диплома:

```text
service_test_assets/videos/single_valid_video.mp4
service_test_assets/archives/multi_valid_videos.zip
```

Файлы видео и ZIP не добавляются в Git, потому что такие данные обычно большие и локальные. В репозитории хранится только структура папок и инструкция.

## 4. Автоматический сервисный прогон

Основной запуск из корня проекта:

```bash
python scripts/run_service_tests.py
```

Что делает скрипт:

| № | Проверка | Ожидаемый результат |
| --- | --- | --- |
| 1 | Загружает одиночное видео из `service_test_assets/videos/` | Создается `job_id`, сервер возвращает список видео |
| 2 | Запрашивает первый кадр одиночного видео | Создается и возвращается JPEG-файл |
| 3 | Автоматически строит ROI по размеру кадра | ROI передается в обработку |
| 4 | Запускает обработку одиночного видео | Задание переходит в `processing`, затем в `done` |
| 5 | Получает JSON-отчет по одиночному видео | В отчете есть `overview`, ссылки и корректное число видео |
| 6 | Скачивает ZIP-отчет по одиночному видео | ZIP существует и не пустой |
| 7 | Повторяет тот же сценарий для ZIP-архива с видео | Все видео из архива извлекаются и проходят сервисный сценарий |
| 8 | Загружает неподдерживаемый `.txt` | Сервер возвращает HTTP 400 и понятное сообщение |
| 9 | Загружает ZIP без видео | Сервер возвращает HTTP 400 и понятное сообщение |
| 10 | Проверяет API-контракты UI-кнопок | Кнопки загрузки, обработки, применения ROI и скачивания связаны с нужными API-эндпоинтами |
| 11 | Проверяет API статуса задания | `/api/jobs/{job_id}`, `/status`, `/report` до обработки и неизвестные `job_id` возвращают ожидаемые ответы |
| 12 | Формирует отчет о тестах | Создаются Markdown и JSON в `data/test_reports/` |
| 13 | Печатает итоговую сводку | По каждому пункту отображается `✅` или `❌`, общее число успешных и ошибочных проверок |

По умолчанию ML-пайплайн заменяется детерминированной тестовой заглушкой. Это нужно, чтобы сервисный тест запускался быстро и не требовал GPU, `.npz`-признаков и весов модели. При этом проверяется полный серверный поток: загрузка, кадр, ROI, запуск обработки, статус, отчет, ZIP-скачивание.

## 5. Прогон с настоящим ML-пайплайном

Если нужно проверить не только сервисный слой, но и реальный анализ видео, подготовьте артефакты, перечисленные в `README.md`:

- `outputs/expA_yolo_pose_only_clean/chunk_features/<video_id>.npz`
- `outputs/expB_rgb_only_clean/chunk_features/<video_id>.npz`
- `outputs/rtmw_full_yolo_gate/frame_features/<video_id>.npz`
- `outputs/expC_pose_rgb_fusion_clean/models/model_comparison/best_C6_LateFusion_Gated_BiGRU.pt`
- `fusion_normalizers.npz`

Затем запустите:

```bash
python scripts/run_service_tests.py --real-ml
```

В этом режиме тест использует настоящий `analyze_videos`, поэтому результат зависит от наличия подготовленных признаков, модели, корректных переменных окружения и вычислительных ресурсов. Подробный лог ML-пайплайна по умолчанию скрывается, чтобы в консоли осталась компактная сводка; если нужен полный вывод пайплайна, добавьте `--verbose-ml`.

## 6. Дополнительные параметры запуска

```bash
python scripts/run_service_tests.py --clean-data
```

Удаляет `data/uploads` и `data/outputs` перед прогоном, чтобы результаты предыдущих запусков не мешали проверке.

```bash
python scripts/run_service_tests.py --report-dir /path/to/reports
```

Задает папку для Markdown/JSON-отчетов о сервисном прогоне. По умолчанию отчеты сохраняются в `data/test_reports/`.

```bash
python scripts/run_service_tests.py --assets-dir /path/to/assets
```

Позволяет использовать другую папку с такой же структурой `videos/` и `archives/`.

```bash
python scripts/run_service_tests.py --real-ml --verbose-ml
```

Показывает полный stdout/stderr настоящего ML-пайплайна. Без этого флага runner скрывает подробные таблицы, progress bar и служебные сообщения пайплайна, а в консоль выводит только пункты проверки и итоговую сводку. Если обработка завершится ошибкой, runner выведет последние строки скрытого ML-лога для диагностики.

Параметры можно объединять:

```bash
python scripts/run_service_tests.py --clean-data --assets-dir /path/to/assets
```

## 7. Модульные тесты

Дополнительно в проекте есть модульные тесты для функций хранения, отчетности, API lifecycle и контрактов frontend-кнопок. Они не требуют тестовых видео, модели и GPU.

Запуск:

```bash
python -m pytest
```

Автоматизированные модульные проверки:

| Модуль | Проверка | Ожидаемый результат |
| --- | --- | --- |
| `app.services.storage.safe_name` | Очистка имени файла от директорий и опасных символов | Имя безопасно для сохранения внутри каталога загрузок |
| `app.services.storage.save_uploads` | Загрузка поддерживаемого видео и игнорирование неподдерживаемых файлов | В список сохраненных попадает только видео |
| `app.services.storage.save_uploads` | Распаковка ZIP с вложенными путями | Извлекаются только видео, имена санитизируются, файлы остаются внутри каталога задания |
| `app.services.reports.enrich_zone_meaning` | Расчет take/return/inspect score | В зонах появляются бизнес-комментарии и агрегированные оценки |
| `app.services.reports.build_job_report` | Формирование отчета по успешным и ошибочным видео | Создаются CSV, XLSX, HTML и ZIP, сводные метрики совпадают с входными данными |
| `app.main` API endpoints | Создание задания, получение кадра, запуск обработки, статус, отчет и скачивание ZIP | Возвращаются ожидаемые статусы, payload и файлы |
| `backend/app/static/app.js` | Контракты кнопок интерфейса с API | Кнопки вызывают нужные API и обрабатывают ответ |

## 8. Как оформить результаты в дипломе

Рекомендуемая структура раздела:

1. Цель и объект тестирования.
2. Описание тестовой среды: ОС, версия Python, версия браузера при ручном просмотре отчета, CPU/GPU, наличие ML-артефактов.
3. Описание тестовых данных: имя одиночного видео, имя ZIP-архива, количество видео внутри архива.
4. Таблица автоматического сервисного прогона с пунктами из раздела 4 и фактическим результатом.
5. Таблица модульных тестов с результатом `passed`.
6. Вывод: какие части сервиса проверены автоматически и какие ограничения остаются.

Шаблон таблицы для сервисного прогона:

| № | Проверка | Фактический результат | Статус |
| --- | --- | --- | --- |
| 1 | Загрузка одиночного видео | `job_id` создан, видео найдено | Успешно |
| 2 | Получение первого кадра | JPEG-файл создан | Успешно |
| 3 | Обработка и отчет | Статус `done`, ZIP-отчет скачан | Успешно |
| 4 | ZIP-архив с видео | Видео извлечены, отчет создан | Успешно |
| 5 | Неподдерживаемый файл | HTTP 400 | Успешно |
| 6 | ZIP без видео | HTTP 400 | Успешно |

Пример вывода:

> Автоматическое тестирование подтвердило работоспособность основных сервисных сценариев: загрузка одиночного видео, обработка ZIP-архива с видео, получение первого кадра, запуск обработки, формирование отчета и скачивание ZIP-архива с результатами. Негативные проверки показали, что неподдерживаемые файлы и архивы без видео отклоняются с понятной ошибкой. Модульные тесты дополнительно подтвердили корректность функций хранения и генерации отчетов.
